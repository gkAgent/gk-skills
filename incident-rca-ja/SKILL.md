# Incident Report & RCA — Japanese SI Format

> 障害報告書と根本原因分析を日本SI標準フォーマットで生成する。
> 顧客報告・社内レビュー・再発防止計画まで一気通貫。

---

## Phase 0: Scope

Ask the user:

> 障害の概要を教えてください。
>
> **障害種別**: システム停止 / 機能不全 / データ不整合 / 性能劣化 / セキュリティインシデント
> **発生日時**: YYYY-MM-DD HH:MM（JST）
> **影響範囲**: 対象システム・影響ユーザー数・業務影響
> **現状**: 復旧済み / 対応中 / 調査中
> **暫定対応**: 実施済みの対応内容

---

## 障害報告書テンプレート

```markdown
# 障害報告書

**報告書番号**: INC-YYYYMMDD-001
**作成日**: YYYY-MM-DD
**報告者**: [担当者名]
**承認者**: [リーダー名]
**顧客報告**: 要 / 不要

---

## 1. 障害概要

| 項目 | 内容 |
|---|---|
| 障害名称 | [例: 受注処理バッチ停止障害] |
| 障害種別 | システム停止 / 機能不全 / データ不整合 / 性能劣化 |
| 発生日時 | YYYY-MM-DD HH:MM（JST） |
| 復旧日時 | YYYY-MM-DD HH:MM（JST） |
| 障害継続時間 | X時間 Y分 |
| 影響システム | [例: 受注管理システム（本番環境）] |
| 影響ユーザー数 | [例: 全社ユーザー 150名] |
| 業務影響 | [例: 当日の受注処理が X件 未処理。翌日補完処理で対応] |
| 重要度 | 高（業務停止）/ 中（一部機能不全）/ 低（影響限定的） |

---

## 2. 障害タイムライン

| 日時 | 出来事 | 対応者 |
|---|---|---|
| HH:MM | [例: 監視アラート発報（バッチジョブ失敗）] | 監視担当 |
| HH:MM | [例: 担当 SE に連絡。調査開始] | SE 田中 |
| HH:MM | [例: エラーログ確認。ORA-01031: insufficient privileges 確認] | SE 田中 |
| HH:MM | [例: DB 管理者に権限付与を依頼] | SE 田中 → DB チーム |
| HH:MM | [例: 権限付与完了。バッチ再実行] | DB チーム |
| HH:MM | [例: バッチ正常終了を確認。復旧完了] | SE 田中 |
| HH:MM | [例: 顧客報告（メール）送付] | PM 鈴木 |

---

## 3. 原因分析（5 Why）

### 直接原因
[例: バッチ処理ユーザー（BATCH_USER）が対象テーブルへの SELECT 権限を持っていなかった]

### 5 Why 分析

| Why | 問い | 答え |
|---|---|---|
| Why 1 | なぜバッチが失敗したか | BATCH_USER に SALES_ORDER テーブルの SELECT 権限がなかった |
| Why 2 | なぜ権限がなかったか | 3/15 のテーブル追加時に BATCH_USER への権限付与が漏れた |
| Why 3 | なぜ漏れたか | テーブル追加の手順書に「アプリユーザーへの権限付与」チェック項目がなかった |
| Why 4 | なぜチェック項目がなかったか | 手順書が整備されておらず、担当者の経験則に依存していた |
| Why 5 | なぜ手順書がなかったか | DB 変更作業の標準化がされていなかった |

### 根本原因
**DB 変更作業（テーブル追加・変更）の標準手順書が存在せず、担当者の経験に依存していたため、権限付与が漏れた**

---

## 4. 対応内容

### 暫定対応（済み）
| 対応 | 実施日時 | 実施者 | 状態 |
|---|---|---|---|
| BATCH_USER に SALES_ORDER の SELECT 権限を付与 | YYYY-MM-DD HH:MM | DB チーム | 完了 |
| バッチジョブを手動再実行 | YYYY-MM-DD HH:MM | SE 田中 | 完了 |

### 恒久対応（予定）
| 対応 | 期限 | 担当者 | 優先度 |
|---|---|---|---|
| DB 変更作業チェックリスト作成（権限付与を必須項目に追加） | YYYY-MM-DD | TL 鈴木 | 高 |
| 既存テーブル全件の BATCH_USER 権限確認と一括付与 | YYYY-MM-DD | DB チーム | 高 |
| 権限不足エラーの監視アラート追加（ORA-01031 → 即時通知） | YYYY-MM-DD | インフラ担当 | 中 |
| バッチ開始前の権限チェックスクリプト導入 | YYYY-MM-DD | SE 田中 | 中 |

---

## 5. 再発防止策

| カテゴリ | 対策 | 期限 |
|---|---|---|
| **プロセス改善** | DB 変更作業チェックリスト整備・レビュー必須化 | YYYY-MM-DD |
| **自動化** | CI/CD で DB 権限チェックを実行（権限不足なら deploy 失敗） | YYYY-MM-DD |
| **監視強化** | バッチ開始〜終了の監視アラート強化 | YYYY-MM-DD |
| **ドキュメント** | DB 変更手順書を Confluence に整備 | YYYY-MM-DD |

## 障害種別ごとの分析観点

| 障害種別 | 追加チェック項目 |
|---|---|
| バッチ停止 | ジョブスケジューラの設定 / 前処理依存 / タイムアウト設定 |
| データ不整合 | トランザクション境界 / 並行処理 / 外部キー制約 |
| 性能劣化 | スロークエリ / インデックス / 接続プール枯渇 / メモリリーク |
| 認証エラー | DB 権限 / SSL 証明書期限 / パスワード期限 |
| セキュリティ | アクセスログ / 不審IPの特定 / 影響範囲の確定（情報漏洩有無） |

---

## Forbidden

- **根本原因を「ヒューマンエラー」で終わらせる禁止** — 「なぜヒューマンエラーが起きたか」まで掘り下げる（Why 4〜5まで）
- **暫定対応と恒久対応の混同禁止** — 区別して記載。暫定対応のみで終わらせない
- **再発防止策なし禁止** — 必ず具体的なアクション・期限・担当者を記載
- **途中停止禁止** / **AI voice禁止**
